# KLC-052 — Design (Option A, operator-confirmed)

> Approach is fixed: **Option A — orchestrator skill in the plugin**
> (`/klc:run <KEY>`). This document is the concrete design and closes the
> two items Q-004 left open. Option comparison lives in
> [`design/options.md`](design/options.md); this file is the build-facing
> design that `impl-plan.md` is derived from.

## 1. Closing Q-004 item 1 — the single phase→agent resolver

### Decision (D-001) — resolver module & function

> [!DECISION D-001]
> The resolver is a **new pure module `core/skills/phase_resolver.py`**
> exposing one function:
>
> `resolve_phase(ticket, phase_id) -> ResolvedPhase`
>
> returning a dataclass `ResolvedPhase(phase_id, track, prompt_path,
> model, cc_model, agent_type, runs_inline, interactive)`. It is the
> single source of truth that BOTH executors consume. No new YAML map is
> introduced — every field is derived from the three existing sources of
> truth, satisfying C-001/C-002.

**Why a Python module and not the SKILL.md or a new CLI verb.** C-001
forbids orchestration logic that duplicates `klc` CLI logic, and the
resolver must be *testable in Python* (the test plan asserts the
XS-vs-subagent decision derives only from `meta.json:track` +
`phases.yml:tracks`, with no hand-kept map). A prompt cannot be unit
tested; a new CLI verb is the rejected Option B. A library module called
by both executors is the minimal seam. The `/klc:run` SKILL.md invokes it
through a thin `python3 -m` shim (no new lifecycle verb — C-001 honored).

### What each field is derived from (no parallel map)

| Field | Source of truth | How |
|-------|-----------------|-----|
| `track` | `meta.json:track` | read ticket meta |
| `prompt_path` | `phases.yml` `work.prompt` | `phases.by_id(phase_id).prompt` (FACT src=core/skills/phases.py:101 verified=2026-06-26) |
| `runs_inline` | `phases.yml:tracks` + `meta.track` | `track == "XS"` AND phase is on the XS fast-track via `phases.track_phases("XS")` (FACT src=core/skills/phases.py:107 verified=2026-06-26) |
| `model` / `cc_model` | `models.yml` | `load_models().resolve(phase_id, track=track).model`, then `plugin_gen._cc_alias(model)` (FACT src=core/skills/models.py:114, core/skills/plugin_gen.py:41 verified=2026-06-26) |
| `agent_type` | generated `klc-plugin/agents/` set | `f"klc-{phase_id}"` only if `klc-plugin/agents/{phase_id}.md` exists; else `None` (inline-only / no subagent). The directory is the authoritative set, generated by `plugin_gen.generate_agents` (FACT src=core/skills/plugin_gen.py:45-115 verified=2026-06-26) |
| `interactive` | `phases.yml` ack picks + a new `work.interactive` flag | `True` when the phase requires the interactive channel (clarify) — see §3 |

> [!FACT] the CC model alias map already exists and must be the single
> alias authority so the resolver returns the same alias `plugin_gen`
> writes into agent frontmatter. src=core/skills/plugin_gen.py:28-42 verified=2026-06-26

### How each executor calls it

```
                       core/skills/phase_resolver.py
                       resolve_phase(ticket, phase_id) -> ResolvedPhase
                          ▲                                   ▲
        consumes (in-session)                      consumes (headless)
                          │                                   │
   klc-plugin/skills/run/SKILL.md            core/skills/runner.py
   (main agent / Task tool)                  (subprocess, claude --print)
```

- **Task-tool path (`/klc:run`, attended).** The SKILL.md runs
  `python3 core/skills/phase_resolver.py --ticket <KEY> --phase <pid> --json`
  to get the `ResolvedPhase` as JSON. If `runs_inline` → the main agent
  does the phase work inline. Else → it dispatches the `klc-<phase>`
  subagent via the Task tool. The pinned `model:` frontmatter on the
  generated subagent is honored by Claude Code natively (C-002), so the
  orchestrator never passes a model itself — it only passes `agent_type`.
- **Headless path (`runner.py`, unattended).** `run_agent()` already
  resolves model + prompt itself; it is refactored to call
  `phase_resolver.resolve_phase` for `prompt_path` + `model` instead of
  resolving them inline, killing the drift between the two paths. The
  hard rule (C-005) is enforced *here*: if `interactive` is true,
  `run_agent` PARKS (writes a park marker, returns a distinct non-zero
  rc) and never dispatches — see §4.

### Drift killed

There is exactly one place that maps `phase_id → (prompt, model,
agent)`. The generated `agents/` directory is the agent-set authority;
`phases.yml` is the prompt/track authority; `models.yml` is the model
authority. The resolver composes them and is the only consumer-facing
surface. (Neutralizes the "second source of truth" con of Option A.)

## 2. Closing Q-004 item 2 — budget-guard + token-telemetry on the Task path

### Decision (D-002) — split the helpers; advise on the Task path, do not enforce

> [!DECISION D-002]
> Extract the three runner-private helpers
> (`_load_budget_limits`, `_estimate_tokens`, `_write_token_metrics`)
> into a new pure module `core/skills/budget_guard.py` with public names
> `load_budget_limits()`, `estimate_tokens(text)`,
> `write_token_metrics(...)`, and a new `check_prompt_budget(track,
> estimated) -> BudgetVerdict`. `runner.py` imports them (behavior
> unchanged). The Task-tool path gets **advisory budget pre-checks** and
> **best-effort telemetry**, NOT hard enforcement — with the rationale
> below.

**Budget guard — advisory on the Task path, hard on the headless path.**
The headless guard works because `runner.py` owns the subprocess and can
abort *before* spending tokens (FACT src=core/skills/runner.py:330-348
verified=2026-06-26). On the Task path the main agent does not control
its own context as a measurable pre-flight string — the inputs are the
ticket's artifacts the subagent reads itself. So the orchestrator calls
`budget_guard.check_prompt_budget(track, estimate_tokens(inputs_blob))`
as a **pre-dispatch advisory**: on a hard-limit breach it adds a
`blocking_question` ("context for <phase> ~N tokens exceeds <track> hard
limit — split the ticket or upgrade track?") and STOPS (reusing the AC-5
stop path); on a soft breach it surfaces a warning and proceeds. This
keeps the *same config* (`config/budgets.yml`) governing both paths
without duplicating limits.

**Token telemetry — best-effort via the structured signal.** The
orchestrator cannot read the subagent's provider usage block (Claude
Code does not expose it to the Task caller). So:
- the structured completion signal MAY carry an optional
  `tokens: {in, out}` block when the subagent can self-report;
- the orchestrator writes it via `budget_guard.write_token_metrics(...,
  source="reported")` (a new source value distinct from `provider` /
  `estimated`);
- when absent, telemetry for that phase is simply omitted on the Task
  path (it is non-fatal today too — FACT src=core/skills/runner.py:140-141
  verified=2026-06-26). Telemetry is observability, not a gate, so
  omission cannot affect correctness.

> [!ASSUMPTION A-001] Claude Code does not surface per-Task token usage
> to the dispatching agent. if-false=if usage becomes available, the
> orchestrator can read it directly and the optional `tokens` signal
> field becomes redundant (no contract break — the field stays optional).

**Why not hard-enforce on the Task path.** Hard enforcement requires
aborting before spend; the orchestrator cannot measure or interrupt the
subagent's actual context. Faking a hard limit from the inputs blob
would be a *different* number than the subagent really consumes — a
false gate. Advisory-on-breach + stop is the honest, fail-safe behavior.

## 3. The `/klc:run` orchestration loop (Deliverable 1)

`klc-plugin/skills/run/SKILL.md` instructs the **main agent** to run this
loop (no Python loop driver — C-001 keeps logic out of the plugin; the
SKILL.md is a prompt, decisions come from the resolver + `phases.yml`):

```
LOOP:
  1. status  = `klc status <KEY> --json`         → {phase_id, state, track}
  2. if state == archived: report DONE, exit.
  3. resolved = `phase_resolver --ticket <KEY> --phase <phase_id> --json`
  4. if resolved.interactive:                    # clarify gate, picks, decisions
        STOP — surface to human (this is a human-interaction point, AC-5/AC-11).
  5. if state == "work":
        budget = budget_guard.check_prompt_budget(...)   # §2 advisory
        if budget.hard_breach: append blocking_question; STOP.
        if resolved.runs_inline:                  # XS fast-track → inline (AC-2)
            do the phase work inline in the main loop.
            signal = {phase, signal:"done", artifacts, blocking_questions, next_action}
        else:                                      # M/L → subagent (AC-2)
            result = Task(subagent_type = resolved.agent_type, prompt = dispatch prompt)
            signal = parse_signal(result)          # AC-3
            if signal is unparseable OR missing required keys:
                retry the SAME phase ONCE (AC-6); second failure → STOP & surface.
        if signal.blocking_questions (non-empty, non-blank): STOP (AC-5).
  6. advance: `klc ack <KEY> --auto`              # AC-4 throttle (KLC-045)
        if rc != 0 (pick_required / gate dirty / ambiguous): STOP & surface (AC-5).
     `klc next <KEY>`
  7. goto LOOP.
```

### Structured completion signal contract (AC-3) — D-003

> [!DECISION D-003]
> Every `klc-<phase>` subagent (and the inline path) MUST end its output
> with exactly one fenced JSON object on its own, as the LAST block:
>
> ```json
> {"phase":"design","signal":"done","artifacts":["design/options.md","impl-plan.md"],"blocking_questions":[],"next_action":"ack"}
> ```
>
> Fields: `phase` (string, must equal the dispatched phase_id),
> `signal` (`"done"|"blocked"|"failed"`), `artifacts` (string[], paths
> relative to ticket dir), `blocking_questions` (string[]; blank/empty
> entries are ignored — edge case), `next_action`
> (`"ack"|"clarify"|"stop"`). Optional: `tokens:{in,out}` (§2).

**Parsing rules (orchestrator).**
- Take the last fenced ```json block; `json.loads` it.
- **Unparseable, or any required key missing, or `phase` mismatch, or
  `next_action`/`signal` not in the allowed set** → treat as a failed
  signal → retry path (AC-6). A `next_action` pointing at a non-existent
  phase is a failure, not a jump (edge case).
- A clean signal is sufficient state: the orchestrator **does not re-read
  the named artifacts** (AC-3, keeps context small). The artifacts list
  is for the human/log, not for the loop to reconstruct state.

This contract is added to `core/agents/*.md` once via a shared
"Completion signal" block (so `plugin_gen` regenerates every
`klc-<phase>` subagent with it — single edit, no per-agent drift).

### Route-aware dispatch (AC-2)

`runs_inline` from the resolver is the only switch: `track == "XS"` and
the phase is XS-eligible → inline; otherwise → subagent. The decision
reads `meta.json:track` against `phases.yml:tracks` only (no hand map).
Edge case: a phase whose `tracks:` lists both XS and M, on an M-track
ticket → `runs_inline=False` (track wins, not phase membership).

### Failure handling (AC-6) — retry once, then stop

Per-phase failure counter lives in the loop (not persisted): a dead
subagent / timeout / unparseable-or-mismatched signal increments it;
`== 1` → re-dispatch the same phase; `>= 2` → STOP and surface (never
advance, never skip). Two consecutive timeouts both surface (edge case).

### Stop conditions (AC-5) consolidated

The loop halts and hands to the human on any of: (a) `resolved.interactive`
phase reached (clarify/picks/decision gates — §4); (b) `ack --auto` rc≠0
(`pick_required` / gate dirty / ambiguous pick — most KLC picks are
`gate: decision`, FACT src=config/phases.yml design/discovery picks,
core/skills/gate_policy.py:74-80 verified=2026-06-26, so the loop
naturally pauses at every human decision); (c) non-empty
`blocking_questions[]`; (d) budget hard breach (§2); (e) second
consecutive phase failure (AC-6).

## 4. Mandatory intake clarify gate (Deliverable 2)

### Where the gate fires — D-004

> [!DECISION D-004]
> The gate is wired in **two coordinated places**:
> (1) `core/phases/intake.py` STAMPS the requirement on low confidence
> (it cannot ask — no interactive channel), and (2) the `/klc:run`
> main loop FIRES the interactive clarify when it sees that stamp. The
> stamp is `meta.json:clarify_required = true`, set whenever
> `route_confidence == "low"` (FACT src=core/skills/route_heuristic.py:179,
> core/phases/intake.py:203-227 verified=2026-06-26).

**Why a stamp, not inline asking.** `intake.py` runs as a CLI subprocess
with no `AskUserQuestion` channel (same constraint as a background
subagent — that is the bug this ticket fixes). It can only *record* that
clarify is owed. Only the main loop has the interactive channel, so it
owns the asking (AC-8). This also makes the gate observable/testable
without an interactive harness: a test asserts the stamp is written on
low confidence (AC-7) and absent on high (AC-7 negative).

**Mandatory, fail-closed (AC-7/AC-10).** When the loop reaches a ticket
with `clarify_required == true` it MUST fire the clarify pass before any
discovery dispatch — unconditional on confidence remaining low, and
unconditional on whether `raw.md` has content (it never pre-screens
content — AC-7). The gate is cleared only by an answered
`AskUserQuestion`; "nothing to add" is a valid answer that clears it
(AC-10). Mandatory = the gate always *fires*, not that the user must
produce content.

### The clarify pass (AC-8/AC-9) — D-005

> [!DECISION D-005]
> The clarify pass reuses `core/agents/intake-triage.md`'s classify +
> enrich machinery (AC-9). `intake-triage.md` is changed from "opt-in,
> non-interactive" to carry an **interactive clarify section** the main
> loop executes: one `AskUserQuestion` with 2–4 batched, design-changing
> questions (AC-8). The questions come from the triage's
> `missing_info[]` (FACT src=core/agents/intake-triage.md:49-54
> verified=2026-06-26).

Flow: triage proposes `missing_info` → main loop issues ONE
`AskUserQuestion` (batch) carrying 2–4 of them → answers are written
back into `raw.md` under the existing `<!-- BEGIN: intake-notes -->`
markers (AC-9, reuse, FACT src=core/agents/intake-triage.md:39-44
verified=2026-06-26) → the route is recomputed by re-running
`route_heuristic.classify` and updating `meta.json` (AC-9) → the
`clarify_required` stamp is cleared. Then discovery's **author** subagent
runs on the enriched input (AC-11 split).

### Discovery split (AC-11)

Discovery is modeled as `clarify (main-loop, interactive)` + `author
(background subagent, synthesis)`. The clarify step is exactly the gate
above; the loop STOPS at it (it sets `interactive=True` via the stamp),
then the human clears it, then `author` (the existing `klc-discovery`
subagent on enriched `raw.md`) runs. No new discovery phase id is
added — the split is a behavior of the loop + the stamp, not a
`phases.yml` reshape (a non-goal: "Changing the phases.yml state set").

### clarify.style config (AC-12) — D-006

> [!DECISION D-006]
> New config file **`config/clarify.yml`** (project override at
> `.klc/config/clarify.yml`, mirroring the profile.yml override pattern,
> FACT src=core/skills/profile-resolve.py:42-50 verified=2026-06-26):
>
> ```yaml
> clarify:
>   style: batch   # batch | serial. GLOBAL. default = batch.
> ```
>
> Loaded by a new `core/skills/clarify_config.py`
> (`load_clarify_style() -> str`). **Fail-closed**: an unknown value is
> rejected at load with a visible error (edge case) — never a silent
> fallback to `batch`. The key is global; any per-track key is ignored
> (AC-12). It governs ONLY the in-client interactive main-loop path; the
> headless runner (parks) and the manual-CLI path (human edits raw.md)
> never read it (AC-12, C-006).

`batch` → one `AskUserQuestion` with all 2–4 questions. `serial` → one
prose question per exchange (superpowers-style). Default-when-absent is
`batch`.

## 5. Dependency impact

`dependency-impact: unavailable (modules.json has no depended_by reverse
edges; blast_radius.available=false)` — FACT src=.klc/index/modules.json,
.klc/tickets/KLC-052/meta.json:77-80 verified=2026-06-26. Fallback:
manual call-site reasoning per touched symbol.

- **New modules, no inbound edges yet**: `phase_resolver.py`,
  `budget_guard.py`, `clarify_config.py`. New leaf modules — additive,
  no existing edge inverted.
- **`runner.py`** gains an edge → `phase_resolver` and → `budget_guard`
  (it already contains the budget logic being extracted; this is a
  *move*, behavior preserved). New coupling `runner → phase_resolver` is
  intentional (single source of truth) and triggers the ADR check.
- **`intake.py`** gains one field write (`clarify_required`); no new
  module edge. Existing intake routing tests must stay green
  (regression: tests/test_intake_routing.py).
- **`plugin_gen._cc_alias`** becomes consumed by `phase_resolver`
  (import). It is currently module-private (`_cc_alias`); the resolver
  importing a `_`-prefixed name is acceptable intra-`core/skills` but the
  cleaner move is to expose `cc_alias` publicly — handled in step-1.
- **Generated `klc-plugin/agents/*`** all change (shared completion-signal
  block added to `core/agents/*.md` then regenerated). Regression:
  model frontmatter must not change (C-002, tests in plugin_gen suite).

No cycles introduced (new modules are leaves; `runner`/`intake` depend on
them, not vice versa).

## 6. ADR

`ADR_NEEDED=yes` — triggers: cross-module boundary crossed (`runner.py`
takes a new dependency on `phase_resolver`; a new `core/skills` seam is
introduced) and a layer boundary is crossed (config↔code: new
`config/clarify.yml` + loader). The ADR records the
**one-resolver/two-executors** architecture and the
**advisory-budget/best-effort-telemetry** decision on the Task path so
future work does not re-litigate D-001/D-002. See
[`design/adr.md`](design/adr.md).
