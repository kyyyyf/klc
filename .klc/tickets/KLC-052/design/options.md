# KLC-052 — Design options: main-agent orchestrator + per-status subagents

Context: today the klc-plugin slash commands only drive **state
transitions** (each `SKILL.md` shells out to `klc <verb>` —
`klc-plugin/skills/*/SKILL.md`). The per-phase subagents in
`klc-plugin/agents/klc-<phase>.md` exist (generated by
`core/skills/plugin_gen.py`) but nothing wires "ticket is in phase X →
dispatch subagent klc-X to do the work". The human invokes each phase
manually, or the headless `core/skills/runner.py` shells out to
`claude --print` per phase. The plugin README already *declares* the
target division of labor ("Execution surface" table) but it is not
implemented as an orchestration loop.

This ticket asks the **main CC agent to become the orchestrator**:
read a ticket's current status, dispatch the matching per-status
subagent (via the Task tool), collect its result/completion signal,
and advance the lifecycle.

---

## Option A — Orchestrator skill in the plugin (main agent drives Task dispatch)

**Summary.** Add a new plugin skill (e.g. `/klc:run <KEY>`) whose
SKILL.md instructs the **main agent** to: (1) `klc status <KEY>` to read
the current phase, (2) look up the phase→subagent mapping, (3) dispatch
the matching `klc-<phase>` subagent with the Task tool, (4) read the
subagent's completion signal, (5) run `klc ack`/`klc next` to advance,
and loop until a human-gated stop (ack pick required, or a blocking
QUESTION). The main agent stays in the CC main loop and acts as the
conductor; each phase's reasoning happens inside its pinned subagent.

**Pros.**
- Matches the plugin's stated execution surface (README table) with no
  change to the headless `runner.py` path — both paths coexist.
- Uses native CC primitives (Task tool, subagent model frontmatter) so
  per-phase model pinning from `models.yml` is honored automatically
  via `plugin_gen.py`.
- Keeps human ack gates intact: orchestrator stops at any
  `pick_required` phase, preserving the existing approval semantics.
- Cheapest to build — mostly a prompt/skill plus a phase→agent map.

**Cons.**
- Orchestration logic lives in a prompt (the SKILL.md), so it is only
  as reliable as the main agent's adherence; needs explicit stop/loop
  guards to avoid runaway auto-advance.
- The phase→subagent map is a second source of truth unless derived
  from `phases.yml` + the generated `agents/` set.
- Risk of the orchestrator skipping the model-handoff guard each phase
  prompt expects; must delegate that to the subagent, not absorb it.

## Option B — Deterministic orchestrator command in the klc CLI

**Summary.** Extend the `klc` CLI with an orchestrating verb that, given
a ticket, emits the *next action* (which subagent to dispatch + its
prompt path) as structured output, and the plugin skill simply relays
that to the Task tool. The CLI owns the phase→agent mapping (derived
from `phases.yml`), the plugin stays thin.

**Pros.**
- Single source of truth for phase→agent in the CLI / `phases.yml`; no
  drift between plugin map and state machine.
- The dispatch decision is testable in Python (unit tests), not just in
  prompt behavior.
- Plugin remains a thin relay, consistent with its current design
  philosophy ("no logic in the plugin").

**Cons.**
- Larger change surface in `core/` (new verb, output contract) plus the
  plugin glue; more code and tests than Option A.
- Still needs the main agent to actually call the Task tool, so it does
  not remove the prompt-reliability concern entirely — it only narrows
  it to "dispatch what the CLI said".
- Couples lifecycle advancement to a new CLI contract that the headless
  runner must also respect to avoid a third dispatch path.

## Option C — Status quo + documentation only (reject)

**Summary.** Keep manual per-phase invocation; document the recommended
"run the matching subagent for the current phase" workflow without an
orchestrator.

**Pros.**
- Zero implementation risk; nothing to maintain.

**Cons.**
- Does not satisfy the ticket — the ask is explicitly an orchestrator,
  not docs.
- Leaves the README's declared execution surface aspirational.
- Listed only to bound the decision space.

---

## Decision (operator-confirmed)

**Picked: Option A** — a thin `klc-plugin/skills/run/SKILL.md`
(`/klc:run <KEY>`) that drives the loop in the main-loop prompt and
dispatches each phase's `klc-<phase>` subagent via the Task tool. No new
CLI verb.

Resolutions that fixed the choice:
- Q-001: "statuses" = lifecycle phases; one `klc-<phase>` subagent per
  phase (the existing generated agents).
- Q-002: loop scope = **run-to-gate** — auto `ack` + `next` across
  phases until the nearest `pick_required` human gate or a blocking
  QUESTION.
- Q-003: build approach = Option A (plugin skill), not the CLI verb.

The map-drift con of Option A is neutralized by deriving the
phase→agent set from the generated `klc-plugin/agents/` directory (and
`phases.yml`) rather than a hand-kept parallel map — borrowing Option
B's single-source-of-truth benefit without its CLI-contract cost.

### Orchestrator design specifics (Deliverable 1)

- **Route-aware dispatch.** XS fast-track phases run **inline** in the
  orchestrator; per-subagent overhead is not worth it for XS. M/L phases
  dispatch the matching `klc-<phase>` subagent via the Task tool. The
  decision reads `meta.json:track` against the phase's `tracks:` in
  `phases.yml`.
- **Structured completion signal.** Subagents return
  `{phase, signal, artifacts[], blocking_questions[], next_action}` so
  the orchestrator parses one small object instead of re-reading
  artifacts — this keeps the main-loop context tiny across many phases.
- **Throttle = existing gate-policy.** Advancement is
  `klc ack --auto` (KLC-045 gate-policy: advances only on clean signals
  + unambiguous forward pick) then `klc next`. No new throttle is built
  (`core/phases/ack.py:170-179`, `core/skills/gate_policy.py:74`).
- **Failure handling.** Dead subagent / timeout / unparseable signal →
  retry the phase once; second failure → STOP and surface to the human.
  Never advance on failure, never silently skip.

## Deliverable 2 — Mandatory intake clarify gate: main-loop vs subagent

The clarify pass must run in the **main loop**, not a background
subagent. This is not a preference — it is a hard capability constraint.

**Why a background subagent cannot do it.** `AskUserQuestion` requires an
interactive channel to the user. A background subagent has none: it runs
detached, its only output is its final message. That is *exactly* why
this discovery run could not ask the operator and instead emitted an
oversized spec from a 3-word ticket. Putting clarify in a subagent would
reproduce the original failure mode.

**Why the main loop.** The main agent owns the interactive session, so it
can fire one batched `AskUserQuestion` (2–4 high-leverage, design-changing
questions), capture the answers, write them back to `raw.md`, and trigger
a re-route — all before discovery's *authoring* subagent runs on enriched
input.

**Why mandatory on low confidence.** `route_heuristic` already flags
`route_confidence == "low"` for short, no-signal tickets
(`core/skills/route_heuristic.py:168-180`). Making the gate mandatory
(rather than opt-in triage) guarantees the cheap clarification always
precedes expensive discovery. The no-op escape ("nothing to add")
preserves user agency without weakening the gate — mandatory means the
gate always *fires*, not that the user must produce content.

**Reuse.** The clarify pass reuses `core/agents/intake-triage.md`'s
classify+enrich machinery rather than introducing a parallel agent; the
change is making it mandatory + interactive + main-loop, not rewriting
it.

### Coupling: discovery split

Because clarify is interactive and authoring is synthesis, discovery
splits into `clarify (main-loop)` / `author (background subagent)`. The
orchestrator's run-to-gate loop STOPS at the clarify gate for free —
it is a human-interaction point, which is already a defined stop
condition (a blocking question / human gate). The two deliverables thus
share one mechanism: "the loop pauses wherever the human must speak."
