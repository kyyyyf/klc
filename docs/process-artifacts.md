# Artefact reference

> See [`config/phases.yml`](../config/phases.yml) for the current
> state machine and [`docs/process.md`](process.md) for the process
> overview.

Every ticket lives at `.klc/tickets/<JIRA-KEY>/`. After Learn the
directory moves to `.klc/tickets/archive/<JIRA-KEY>/` unchanged.

Filenames are stable — skills and agents look for them by name.

## File layout (live ticket)

```
.klc/tickets/PROJ-4502/
  meta.json                  # source of truth for phase, track, metrics
  raw.md                     # user's original description (immutable)
  spec.md                    # discovery output (authority: human)
  test-plan.md               # test-planner output
  design/
    options.md               # three options + recommendation
    adr.md                   # optional, only when ADR_NEEDED=yes
  impl-plan.md               # step list; bumped during Build
  build-log.md               # running journal of build iterations (S/M/L)
  build/
    step-N-brief.md          # dependency-resolved context for step N (klc task-brief)
    step-N-impl-report.md    # impl agent's outcome + evidence for step N
    step-N-review-package.md # reviewer input: brief + report + diff
    step-N-review.md         # per-step review findings + verdict
  manual-checklist.md        # only when estimate.manual >= 2
  retrospective.md           # final, human-authority after Learn
  README.md                  # auto-generated summary from ticket-readme.md.j2
  .index.json                # inline-item graph, regenerated by items.py
  scratch/                   # in-session agent notes; archived at integrate
  discovery-context/         # transient bundle; removed at design
  design-context/
  build-context/
```

`archive/PROJ-4502/` keeps everything except the transient
`*-context/` bundles; `scratch/` is renamed to
`archived-scratch-<ts>/` by `scratch.py archive`.

## Per-artefact schema

### `meta.json`

Machine-readable single-source-of-truth:

```json
{
  "ticket":        "PROJ-4502",
  "kind":          "feature | bug | tech | unknown",
  "phase":         "<one of lifecycle.PHASES>",
  "phase_history": [{"phase": "...", "started_at": "...", "finished_at": "..."}],
  "track":         "XS | S | M | L | null",
  "estimate":      {"complexity":0-3,"uncertainty":0-3,"risk":0-3,"manual":0-3,"total":0-12},
  "layer":         "code | content | config | mixed | unknown",
  "affected_modules": ["Auth", "AuthCore"],
  "related_tickets":  ["PROJ-4201"],
  "manual_outcome":   "pass | fail | null",
  "pre_merge_snapshot": {"spec.md": "<hash>", ...},
  "merge_sha":    "<git sha>",
  "merged_at":    "<ISO>",
  "alerts":       [],
  "rework_count": {"build": 1, "review": 0},
  "metrics":      {"intake_ms": 12, "discovery_ms": 18400, ...}
}
```

### `raw.md`

Front-matter then user text. The optional intake-triage agent may insert an
`<!-- BEGIN: intake-notes -->` block listing missing details and a
provisional track; the block is a signal, not a failure.

### `spec.md`

Rendered from `spec.md.j2` (full) or `spec-short.md.j2` (XS track).
Sections: Goals, Problem / Context, Acceptance Criteria, Non-goals,
Constraints, Affected modules, Open questions, Estimate, plus an
opt-in `<!-- BEGIN: manual --> ... <!-- END: manual -->` block that
never gets overwritten by discovery re-runs.

Full form caps: ~80 lines recommended; hard-capped at 120 by
consistency check.

Short form: max 15 lines, exactly what the template emits. No
options / non-goals / rollout sections.

**`Affected` section contract (discovery-lite / XS / S):** every entry
must be either LSP-verified (`src=path:line`, mandatory) or an explicit
assumption (`[!ASSUMPTION if-false=scope-may-expand]`). Unanchored
module names without either marker are not permitted.

### `test-plan.md`

**S-track**: written by `discovery-lite` in the same agent call as
`spec.md` — acceptance coverage table (e2e/acceptance only, no detailed
section). No separate `acceptance-test-plan` phase runs for S.

**M / L track**: written in two passes by `core/agents/test-planner.md`:

- **Phase 2 (acceptance mode, M/L)** — fills `## Acceptance coverage`
  (AC → e2e/acceptance test), `## Edge cases`, `## Regression
  scenarios`, and `## Manual checklist` when `estimate.manual ≥ 2`.
  Leaves `## Detailed coverage` as a `TBD` comment.
- **Phase 4 (detailed mode, M / L only)** — replaces the `TBD`
  comment with a `## Detailed coverage` table keyed to the impl-plan
  step IDs. The acceptance section and the manual block stay
  verbatim.

Rendered from `test-plan.md.j2` (full — both sections) or
`test-plan-short.md.j2` (XS / S — acceptance-only table; no
detailed section).

Authority: `hybrid`. Agent regenerates the generated body across
re-runs; the `<!-- BEGIN: manual -->` block is preserved.

### `design/options.md`

Always full (no short form — designing at all means not XS). Three
options A / B / C with required fields from
`core/agents/design.md` §1. One option carries `recommended: true`.
Last lines emit `ADR_NEEDED=yes|no REASON="..."`.

### `design/adr.md`

Only when `ADR_NEEDED=yes` and the human picked the option. Format
follows MADR via `core/templates/ADR.md.j2` (pre-existing).

### `impl-plan.md`

Authored by the design agent (S-track: by discovery-lite) to match the
shape in `impl-plan.md.j2` (full) / `impl-plan-short.md.j2`. These
templates are a **contract sample**, not a runtime renderer — no code
renders them today.

#### Executable step contract (enforced by `tests/prompt_harness.py`)

Every `## step-N` must include all of the following fields:

| Field | Harness | Description |
|-------|:-------:|-------------|
| `Goal` | ✓ | One sentence — the behaviour or structural change. |
| `RED` | — | Failing test to write first; or `RED: not applicable — <reason>` for prompt/doc/config steps. |
| `GREEN` | — | Minimal code change expected to make RED pass. |
| `VERIFY` | ✓ | Exact test command or suite/case name. |
| `Expected` | ✓ | Expected output of the VERIFY command (e.g. `1 passed`). |
| `COMMIT` | ✓ | Proposed commit subject, prefixed `<ticket-key> step-N:`. |
| `Affected` | ✓ | Concrete file paths. |
| `Interfaces` | ✓ | Function/method signatures added or changed, or `none`. |
| `Depends on` | — | Earlier step IDs required, or `none`. |
| `Code sketch` | ✓ | A non-empty fenced block showing the key change. |

**Harness ✓** = checked by `impl_plan_violations()` in `core/skills/impl_plan_check.py`
(re-exported via `tests/prompt_harness.py`).
**Harness —** = contract convention enforced by review and prompt discipline, not the harness.
`RED`, `GREEN`, and `Depends on` are in the latter category — they are required by the prompts
and reviewed by humans, but not mechanically gated.

**Plan-completeness gate (KLC-036)**: `impl_plan_violations()` is also called by
`core/skills/phase_completion.py` at two ack points:
- `discovery-lite` ack (S-track): if `impl-plan.md` is present, it must be clean.
- `design` ack (M/L-track): `impl-plan.md` is a required output; it must be clean.
A step with a missing required field, a placeholder token outside fenced blocks,
or an empty code fence blocks ack until the plan is corrected.

**`RED: not applicable` exemption**: steps that are prompt/doc/config
only (no behaviour change) may mark `RED: not applicable — <reason>`,
which exempts both `Code sketch` and the non-empty fence check. This is
the **only** sanctioned way to omit a code sketch. The exemption relies
on author discipline; the harness cannot verify whether the stated reason
is genuine, so reviewers should reject any `RED: not applicable` on a
step that clearly changes behaviour.

Full (M/L): step-1 ... step-N, each following the executable contract above.
For M-track, the `detailed-test-plan` agent appends a `**Tests:**` sub-block
to each step (unit/integration test names + target symbols) in place of a
separate detailed coverage phase.
Manual-blocks preserved.

Short (S-track): 1–3 steps, same per-step contract, kept compact.

### `build-log.md`

Append-only journal written by `core/agents/impl.md` during the Build
phase (S/M/L tracks). Each iteration appends one entry:

```markdown
## Step N — <ISO datetime>
**Attempt**: <brief description>
**Outcome**: green | red | blocked
**Notes**: <what changed, DECISION refs if plan diverged>
```

Before `klc ack` the impl agent must also append an `## Evidence` section
with at least one non-empty fenced block (command + pasted output) for each
acceptance check performed. `build:ack` is mechanically blocked without it.

```markdown
## Evidence

```
$ python3 -m pytest tests/ -q
15 passed in 0.42s
```
```

Authority: `generated` (agent appends; never overwritten). Persists
through review rework cycles so reviewer and retrospective agent can
see the full build history. Archived with the ticket.

### `manual-checklist.md`

Emitted by `core/agents/manual-check.md`. Checkboxes match AC
phrasing verbatim; no paraphrasing. Human ticks through; outcome
recorded via `klc manual <key> --continue --outcome=<pass|fail>`.

### `retrospective.md`

Rendered from `retrospective.md.j2`. Citation-heavy: every
observation is either a `[!FACT F-R...]` item with `src=` or a
cross-reference to meta.json / metrics rollup. Proposed allowlist /
deny / few-shot updates — maximum 2 each.

### `README.md`

Generated from `ticket-readme.md.j2` whenever the ticket index is
rebuilt. Single-screen summary, links to every artefact, item
counts, metrics snippet. Useful for humans browsing
`.klc/tickets/` without opening every file.

### `.index.json`

Machine-readable graph of every inline item in every artefact.
Regenerated by `core/skills/items.py index`. Used by
`consistency_check.py` and the retrospective agent.

## Authority model

Each artefact carries `authority` in its frontmatter:

- `human` — never regenerated by an agent. `spec.md` after ack and
  `retrospective.md` after Learn.
- `generated` — overwritten every run. Index files, `options.md`,
  `README.md`.
- `hybrid` — agent regenerates the body, but blocks delimited by
  `<!-- BEGIN: manual --> ... <!-- END: manual -->` are preserved
  verbatim.

Full contract: `process-proposal.md` §4.

## Short-form eligibility

| Track | spec | impl-plan | test-plan |
|-------|------|-----------|-----------|
| XS | short | short | short |
| S  | full  | short | full |
| M  | full  | full  | full |
| L  | full  | full  | full |

A track upgrade (`klc ack ... --upgrade-track`) forces re-render
into the full form on next regeneration.
