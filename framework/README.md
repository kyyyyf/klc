# Codebase Intelligence Framework

Helps Claude (or any agent that can read markdown prompts) work on a large
codebase without reading every file every turn. Produces a stable module
map, per-module `CLAUDE.md` files, and a feature/bug workflow gated by a
validator, a test-first step, and a multi-agent code review.

Works across engines and stacks via **profiles**. The active profile
declares which rules, reviewers, excludes, and module-discovery mode to
use. Ships with `ue` (Unreal Engine) and `generic`.

## Quick start

```bash
./framework/scripts/install-deps.sh          # 1. system deps
cp framework/profiles/ue/mcp.json .mcp.json  # 2. start Serena + ast-grep (pick profile)
./framework/scripts/init.sh                   # 3. scan + dep-graph
                                              # 4. follow the prompts it prints:
                                              #    inventory -> decompose -> docgen
./framework/scripts/init.sh --finalize        # 5. record baseline git sha
```

`./framework/scripts/update.sh` keeps everything in sync after subsequent
commits. `./framework/scripts/feature.sh "desc"` and `bug.sh "desc"` run
the validator → task → test → review → adr flow.

## Picking a profile

Edit `framework/config/profile.yml`:

```yaml
profile: ue        # or: generic, or any directory under framework/profiles/
```

A profile's `manifest.yml` lists:

- `rules` — ast-grep rule directories to load (core + profile-specific).
- `sgconfig` — optional project-level ast-grep config (e.g. route `.h` to cpp).
- `reviewers.always` / `reviewers.conditional` — review sub-agents.
- `excludes` — directories never scanned.
- `module_discovery.mode` — how `source_roots` are computed.
  Supported: `build-cs` (Unreal), `conventional-dirs` (src/lib/pkg/...).
- `content_extensions` — what counts as non-code content for the validator
  (e.g. `.uasset`, `.umap` on UE). Drives the `spec.layer` field.
- `large_project_threshold_files` — agents prefer Serena over ast-grep once
  the file count crosses this.

## Layout

```
framework/
  core/
    agents/             # prompts that never change across profiles
    agents/review/      # sub-agents run for every profile
    skills/             # shell + python skills
    rules/              # ast-grep rules for generic languages
    templates/          # Jinja2 templates (CLAUDE.md, ADR, report)
  profiles/
    ue/
      manifest.yml
      sgconfig.yml      # .h -> cpp
      agents/review/    # ue-conventions, replication, content-pipeline
      rules/cpp-unreal/ # UCLASS/UFUNCTION/UPROPERTY
      mcp.json          # Serena + ast-grep (with sgconfig)
      hooks/            # profile-specific scripts (reserved)
    generic/
      manifest.yml
      mcp.json
  config/
    profile.yml         # which profile is active
    reviewers.yml       # review gates, external reviewer, mutation thresholds
  scripts/              # init / update / feature / bug / review / install-deps
  index/                # generated JSON (gitignored, keep .gitkeep)
  logs/                 # generated logs   (gitignored)
  reports/              # generated review reports
  .last-run             # last indexed git sha
docs/
  adr/                  # ADR files
CLAUDE.md               # root, generated by docgen
```

## MCP servers

| Server   | What it does                                      | Where configured                      |
|----------|---------------------------------------------------|---------------------------------------|
| Serena   | Symbol graph via LSP; persistent cache.           | `.serena/project.yml` (auto-detected) |
| ast-grep | Structural AST search; YAML rules.                | Profile `sgconfig.yml` + `rules/*`    |

Large projects: **Serena is the primary source of truth** for symbols.
Agents only fall back to ast-grep when Serena has no backend for the
language or when a structural-pattern search is easier than an LSP query.
The threshold is `large_project_threshold_files` in the profile manifest;
below it, either tool is fine.

## Flow

| Mode    | Script        | What runs                                             |
|---------|---------------|-------------------------------------------------------|
| init    | `init.sh`     | scan → dep-graph → inventory → decompose → docgen     |
| update  | `update.sh`   | periodic agent on the git diff since last run        |
| feature | `feature.sh`  | validator(feature) → task → test → review → adr?      |
| bug     | `bug.sh`      | validator(bug) → task → test → review → adr?         |

## Design

- **Lazy context.** The task agent calls `context-loader.py` to pick the
  minimum set of `CLAUDE.md` files and public-API signatures. Budget
  target: under 5% of the project's symbol count.
- **Manual notes survive re-generation.** Every `CLAUDE.md` has a
  `<!-- BEGIN: manual --> ... <!-- END: manual -->` block that docgen
  copies through on every run.
- **ADRs are not mandatory.** The task agent signals
  `ADR_NEEDED=yes|no`; the adr agent records only on a real trigger
  (public-API change, new dependency, cross-module change, crossing
  layer boundary like code→content).

## External reviewer (optional)

Edit `framework/config/reviewers.yml`, set
`external_reviewer.enabled: true`, pick provider/model/api_key_env, export
the key, then:

```bash
./framework/scripts/review.sh --diff HEAD --spec path/to/spec.md --external
```
